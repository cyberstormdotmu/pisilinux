Things To Check While Reviewing¶

Last Modified Date: October 26, 2011

Author:Semen Cirit
Version:0.2

In order to review a new package, there are a lot of things to check. The below list only provides some guidelines for new rewievers in order to identify a way to follow. But of course this list is not enough. Reviewers should also use their own experiences when reviewing a package.

  1. The package name must be suitable to package naming.
  2. The package must meet the packaging guidelines.
  3. The package must meet licensing guidelines.
  4. The license tag in the pspec.xml file must match the actual license short names.
  5. The source code of the package and comments must be written in english.
  6. The source code of the package must be legible.
  7. The package must successfully compile and build into pisi for at least one architecture supported.
  8. If the package could not successfully compile, build or work on a specific architecture, then those architectures should be specified in pspec.xml file with ExcludeArch tag.
  9. All build dependencies must be listed in pspex.xml file, except for any that are listed in the dependencies excepted document.
  10. The translations.xml file must be added to package.
  11. Almost every pisi package (or subpackage) have shared library files, you must run checkelf for every package and find broken links.
  12. Packages must not bundle copies of system libraries.
  13. Permissions on files must be set properly. Executables should be set with executable permissions, See Additional Files.
  14. Each package must use related actionsapi modules rather than recreating similar modules in main.
  15. Package must contain code, or permissable content.
  16. If the size or the quantity of the documentation files are large, they must go in a packagename-doc subpackage.
  17. Header files or unversioned shared libraries must be in a packagename-devel subpackage.
  18. Libtool archives .la must not be included in packages, these must be removed in the actions.py if they are built.
  19. The GUI application pacakges must contain packagename.desktop file, and the icon tag of this application should also be defined in pspec.xml file.
  20. The reviewer should test the package in a related proper Pardus system.
  21. The reviewer should test that the application runs as described. For example the applcation should not be crashed or give a segfault.
  22. Usually, subpackages require the base package as a dependency, it should be defined as a strict dependency for subpackages as needed.
  23. Pkgconfig(.pc) situation should be examined and their package placement should be decided.
